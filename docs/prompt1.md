## 사용자 흐름

### 0. 안내

처음 접속하면 다음 취지로 상황을 간단히 설명한다.

`7월 20일 운영한 선린냥이 × HMH 키오스크에서 일부 후발주 주문의 구매자 정보가 저장되지 않았습니다. 해당 주문을 확인하기 위해 기억나는 결제 정보를 입력해 주세요.`

- 조회 대상은 후발주 현금 결제 주문이라는 점
- 기억나지 않는 항목은 공식 Instagram DM으로 문의할 수 있다는 점
- `주문 확인 시작하기` 버튼

### 1. 후발주 구매 여부

`결제 내역에 후발주 상품이 있었나요?`

- 예
- 아니요
- 기억나지 않음

`아니요`를 선택하면 누락 대상 주문이 아닐 가능성이 높다는 안내를 표시하되, 실제 주문 존재 여부는 공개하지 마라.

### 2. 실수령 상품 선택

현장에서 실제로 받아간 상품의 다음 정보를 선택하게 한다.

- 상품
- 옵션
- 수량
- `실수령한 상품 없음`

DB에서 실제 판매 상품과 옵션을 읽어와 표시하되, 내부 ID나 주문 수는 노출하지 마라.

### 3. 후발주 결제 상품 선택

나중에 받기로 결제한 상품을 선택하게 한다.

- 상품
- 옵션
- 수량
- 복수 선택 가능

실수령 상품과 후발주 상품은 명확히 구분한다.

### 4. 최종 결제금액

사용자가 기억하는 실제 현금 결제금액을 직접 입력하게 한다.

- 원 단위 숫자 입력
- 상품 선택에 따른 예상 금액을 자동으로 정답처럼 표시하지 마라
- 할인 적용 후 실제 결제금액을 입력하도록 안내

### 5. 할인 및 쿠폰

- 사용함
- 사용하지 않음
- 기억나지 않음

DB에 저장된 실제 할인·쿠폰 사용 여부와 비교한다.

### 6. 구매 시간대

구매 가능 범위는 `오후 12:00~오후 1:00`이다.

- 사용자가 시간 또는 대략적인 시간대를 선택
- 기억나지 않음 제공
- 사용자가 지정한 시간을 중심으로 최대 ±10분 범위를 선택 가능
- 정확도에 따라 보조 점수 부여
- 초 단위 정확한 시간은 요구하지 마라

예시 점수:

- DB 시간과 ±5분 이내: 30점
- ±10분 이내: 20점
- 사용자가 넓은 범위를 선택했지만 해당 범위에 포함: 10점
- 기억나지 않음: 0점

### 7. 판매 레인

실제 레인 명칭을 조사해 사용자가 이해할 수 있는 표현으로 선택하게 한다.

- 레인 정확히 일치: 30점
- 기억나지 않음: 0점

내부 코드나 UUID를 노출하지 마라.

### 8. 정보 확인

입력 내용을 한 화면에서 정리해 보여주고 수정할 수 있게 한다.

마지막에 다음 취지의 확인을 표시한다.

## 주문 매칭 로직

상품 정보는 점수만 더하지 말고 필수 일치 조건으로 사용한다.

필수 조건:

- 실수령 상품·옵션·수량 일치
- 후발주 상품·옵션·수량 일치
- 최종 결제금액 일치

결제수단은 누락 주문 3건 모두 현금이므로 입력받지 않고 서버에서 현금 주문만 대상으로 제한한다.

보조 점수:

- 판매 레인 일치: 30점
- 쿠폰·할인 여부 일치: 20점
- 구매 시간 정확도: 최대 30점

상품과 금액은 서로 종속될 수 있으므로 중복 점수를 주지 마라.

조회 성공 조건은 임의로 느슨하게 만들지 말고 다음 조건을 기준으로 구현하라.

- 필수 조건 모두 일치
- 보조 점수 40점 이상
- 가장 높은 후보가 하나만 존재
- 두 번째 후보와 20점 이상 차이
- 단순한 1개 상품 주문은 시간대와 판매 레인이 모두 일치해야 함

조건이 부족하거나 후보가 여러 개라면 주문번호나 존재 여부를 노출하지 말고 다음처럼 동일하게 표시한다.

`입력한 정보만으로 주문을 정확히 확인하기 어렵습니다. 기억나지 않는 항목이 있다면 공식 Instagram DM으로 문의해 주세요.`

일치하는 주문이 확실할 때만 기존 영수증 디자인을 사용해 표시한다.

표시 정보:

- SUNRIN CAT
- 주문번호
- 주문일시
- 결제수단: 현금
- 판매 레인
- 실수령 상품
- 후발주 상품
- 각 상품의 상품명·옵션·수량·일반/후발주 여부

이름과 학번은 기존 주문 DB에 없으므로 조회 성공 후 입력받는다. 기존 주문 행에는 직접 저장하지 말고, 신규 확인 신청 테이블 또는 안전한 임시 상태로 별도 저장하라.

버튼:

- `본인 주문이 맞습니다`
- `주문 정보가 다릅니다`
- `Google Form 작성하기`
- `영수증 출력`
- `홈으로`

Google Form 주소는 환경변수 또는 설정 파일로 쉽게 변경할 수 있게 구현하라.

## 개발 방식

1. 먼저 기존 프로젝트와 Supabase 구조를 읽고 구현 계획을 작성한다.
2. 실제 누락 주문 3건을 코드에 하드코딩하지 말고 안전한 서버 조회 조건으로 식별한다.
3. 신규 프로젝트를 구성하고 전체 흐름을 구현한다.
4. 샘플 또는 테스트 데이터로 매칭 로직을 자동 테스트한다.
5. 정상 조회, 오답, 후보 중복, 제한 초과, 직접 URL 접근을 테스트한다.
6. 기존 Supabase 데이터가 변경되지 않았음을 마지막에 검증한다.
7. 실행 방법, 환경변수, 신규 테이블, 보안 설계, 테스트 결과를 README에 기록한다.

모호한 부분은 임의로 기존 데이터를 수정하지 말고, 코드와 DB 구조를 조사한 뒤 가장 안전한 방식으로 구현하라.
